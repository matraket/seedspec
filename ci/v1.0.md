## Generation Rules

### Strict Order
- NO generar documento N sin tener N-1 completo y validado
- Cada documento DEBE referenciar explícitamente sus inputs
- Al completar documento: solicitar confirmación antes de continuar

### Document Structure
```markdown
# [Título]
**Proyecto:** Associated  
**Versión:** X.Y  
**Fecha:** [auto]  
**Inputs:** [lista documentos fuente]  
**Estado:** Borrador | Revisión | Aprobado

---
[Contenido específico del documento]

---
## Changelog
- vX.Y: [cambios]
```

### Traceability
- RF: `RF-XXX` (XXX = 001-999)
- RNF: `RNF-XXX`
- BC: `BC-[Nombre]`
- ADR: `ADR-XXX`
- US: `US-XXX`
- UC: `UC-XXX`

Toda referencia cruzada usa estos IDs.

### DDD Guidelines (Doc 3)
- Bounded Contexts derivados de KB-001 §"Por qué permite demostrar lo aprendido"
- Agregar contexts identificados en KB-002 §12 (necesidades específicas)
- Para cada BC: Aggregates, Entities, Value Objects, Domain Events
- Context Map con relaciones entre BCs

### MVP Scope (Doc 7)
Priorizar según KB-001: "empezar con 2-3 bounded contexts core"
- Core: Membresía, Tesorería
- Supporting: Eventos (simplificado)
- Excluir v1: funcionalidades específicas (cofradías, clubes, etc.)

## Interaction Protocol

### On Session Start
1. Mostrar KB Index actual
2. Mostrar último documento generado
3. Indicar siguiente documento pendiente

### On Document Request
1. Confirmar inputs disponibles
2. Generar estructura + contenido
3. Solicitar revisión/validación
4. Tras aprobación: actualizar índice interno

### Version Bumping
- Nuevo documento generado → minor bump (v1.X)
- Cambio estructural KB → major bump (vX.0)
